3. New Issues Discovered (7)

These issues were either introduced during the fix sprint or became visible now that the foundation has been resolved.

Critical og:image returns 404

All 5 pages reference /images/og-default.jpg in og:image and twitter:image tags, but the file does not exist in /public/images/. Every social share from LinkedIn, X, or WhatsApp will render with no image, degrading CTR.

Fix: Upload a 1200×630px JPG to public/images/og-default.jpg. Use site screenshot or brand graphic. Also add page-specific og:image paths for service pages if available.

Critical /privacy and /terms return 404

Footer links pointing to /privacy and /terms return 404. These pages are required for Google Business Profile verification, PDPA compliance (Singapore), and general E-E-A-T trustworthiness signals. Their absence is a trust flag.

Fix: Create src/pages/privacy.astro and src/pages/terms.astro. Minimal pages are acceptable. Add to sitemap.

High Schema @id cross-referencing broken

The Organization schema uses "@id": "https://innovate-proto.vercel.app/#organization" but Service and FAQPage schemas on inner pages do not reference this @id in their provider or publisher fields. This fragments the Knowledge Graph entity graph and prevents Google from connecting the service pages back to the organization entity.

Fix: In each Service schema, change "provider": {"@type": "Organization", "name": "Innovate"} to "provider": {"@id": "https://innovate-proto.vercel.app/#organization"}.

High Address contradiction — Singapore vs. KL

The homepage and Organization schema state the business is in Singapore. The Contact page and one JSON-LD block state "Kuala Lumpur, Malaysia". Google's entity resolution will flag this contradiction and may suppress Knowledge Panel eligibility.

Fix: Decide the canonical address. Update all references consistently: homepage copy, contact page, and all JSON-LD address fields. If operating in both countries, use a multi-location approach with separate LocalBusiness schemas per location.

High Hero image contradictory loading attributes

The hero <img> element has both loading="lazy" and fetchpriority="high". These are contradictory: lazy defers loading until near-viewport, while fetchpriority=high tells the preload scanner to prioritize immediately. The result is browser-dependent behavior that will hurt LCP on mobile, where the LCP element is typically this image.

Fix: Remove loading="lazy". Keep fetchpriority="high". For the LCP image, also add <link rel="preload" as="image"> in the <head>.

Medium Trailing slash inconsistency

The sitemap outputs URLs with trailing slashes (/about/, /services/) but canonical tags emit without trailing slash (/about, /services). Vercel redirects the trailing-slash variant to no-slash by default, which creates a redirect chain for sitemap-submitted URLs. This wastes crawl budget and dilutes link equity.

Fix: Set trailingSlash: 'never' in astro.config.mjs and update vercel.json to add "trailingSlash": false. Or, prefer trailing slashes by setting trailingSlash: 'always' and updating canonical tags accordingly.

Medium Testimonial author name inconsistency

Two testimonials reference the same person with different names: "John D." on the homepage hero section and "John Doe" in the Services page testimonial block. This E-E-A-T credibility signal is weakened when the same testimonial appears with different identifiers. AI crawlers may flag this as inconsistent or fabricated social proof.

Fix: Standardize testimonial data. Create a single data source (e.g., a content collection in Astro) for testimonials and reference it from all pages to prevent future drift.
